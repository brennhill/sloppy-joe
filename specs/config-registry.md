# Feature: Config Registry

> Generated by `/feature` on 2026-03-28. Review before implementation.

---

## Intent

### 1. What problem does this solve?
Running `sloppy-joe check` across multiple repos requires explicit `--config` or `SLOPPY_JOE_CONFIG` on every invocation. This doesn't scale — each project needs different canonical rules, CI pipelines need the full path, and new projects require manual config file creation + wiring. Even the author finds it annoying.

### 2. How will we know it worked?
`sloppy-joe check` with zero flags resolves the correct per-project config in any registered repo. No `--config`, no env vars, no aliases.

### 3. What is out of scope?
- Config creation/scaffolding (auto-generating ecosystem-specific canonical rules)
- Config syncing across machines
- Config inheritance or merging between global default and per-project configs

### 4. What must NOT happen?
- Config must never be readable from inside the project directory (AI agent attack surface preserved)
- Existing `--config` / `SLOPPY_JOE_CONFIG` workflows must not break (they're in CI pipelines)
- Must not silently use the wrong config or silently run without config
- Must not silently degrade to "no config" — fail closed

### 5. Pre-mortem findings
- **Registry file is writable by the user** — an agent could rewrite it to point at a permissive config. Accepted risk: CI doesn't use the registry (uses explicit `--config`), and an attacker with user-level shell access has bigger problems.
- **Path canonicalization mismatches** — cwd resolves differently than stored path (symlinks, trailing slashes). Mitigated by canonicalizing at both write and read time.
- **Symlink attacks on config dir** — mitigated by existing cache.rs symlink-safe patterns.

### Thinking Record: Intent
**Decided:** Build a registry mapping repo roots to config files, with fail-closed behavior when no config is found.
**Reasoning:** The tool's security model requires config outside the repo. But requiring explicit paths on every invocation creates adoption friction. A registry is explicit (unlike convention-based lookup), survives repo renames (unlike dirname-based), and is unambiguous (unlike cascade).
**Considered and rejected:** eslint-style config cascade (ambiguous for monorepos/worktrees), convention-based `~/.sloppy-joe/<dirname>/config.json` (name collisions, breaks on move).
**Data consulted:** Current config.rs implementation, issue #2 design.
**Assumptions:** Single-user local dev tool; CI always uses explicit `--config`.
**Risks accepted:** Registry file writable by user. Path canonicalization needs care.
**Skipped/declined:** None.

## Behavioral Spec

### User Stories

**Setup (once per repo):**
1. User runs `sloppy-joe init --register` in a repo
2. sloppy-joe creates `{config_home}/sloppy-joe/{dirname}/config.json` with template content
3. sloppy-joe writes the git root → config path mapping to `{config_home}/sloppy-joe/registry.json`
4. User edits the config to add their canonical rules

**Daily use:**
1. User runs `sloppy-joe check` (no flags) in a registered repo
2. sloppy-joe finds git root, looks up registry, loads the mapped config
3. Scan runs with correct per-project rules

**Existing CI use (unchanged):**
1. CI runs `sloppy-joe check --config /path/to/config.json`
2. `--config` takes priority over registry — registry is never consulted

### Mechanism
Explicit registry file (`registry.json`) maps canonicalized git roots to canonicalized config file paths. On `sloppy-joe check`, the tool:
1. Checks `--config` flag (highest priority)
2. Checks `SLOPPY_JOE_CONFIG` env var
3. Finds git root (walk up from cwd to `.git`), looks up in registry
4. Checks for `{config_home}/sloppy-joe/default/config.json` (global fallback)
5. Returns error: "No config found. Run `sloppy-joe register` or pass `--config`."

This works because: CI always uses step 1 or 2 (explicit). Local dev uses step 3 after one-time setup. New users hit step 5 and get a clear action to take.

### States and Transitions

A repo exists in one of three states:
1. **Registered** — registry maps git root to a specific config file
2. **Unregistered with global default** — no registry entry, but `{config_home}/sloppy-joe/default/config.json` exists
3. **Unregistered, no default** — no config available, blocking error on `check`

Transitions:
- `register` → Registered
- `unregister` → Unregistered (falls to state 2 or 3)
- `init --register --global` → creates global default (moves unregistered repos from state 3 to state 2)
- Deleting global default → unregistered repos drop from state 2 to state 3

### Concurrency
Two `register` calls racing on `registry.json` — one could clobber the other's entry. Mitigated by atomic write (temp file + rename), which prevents corruption but not lost entries. Acceptable for a single-user local dev tool — re-register recovers.

### Error Cases
All fail closed:
- **Registry entry points to missing config file** → blocking error with actionable message
- **Resolved config path is inside project directory** → blocking error (existing `ensure_config_outside_project` check applies to registry-resolved paths too)
- **Corrupted/malformed registry.json** → blocking error
- **No registry entry and no global default** → blocking error: "No config found. Run `sloppy-joe register` or pass `--config`."
- **No git root found** (not in a git repo) → blocking error (without registry, need explicit `--config`)
- **Config home directory doesn't exist** → `register` and `init --register` create it

### Thinking Record: Behavioral Spec
**Decided:** Fail-closed on missing config. Registry lookup uses git root. Five-step resolution cascade.
**Reasoning:** Fail-closed is consistent with the tool's security philosophy — silent degradation to "no canonical rules" is a false sense of security. Git root (not cwd) ensures subdirectory runs still match.
**Concurrency risks:** Registry file race on concurrent writes. Atomic write prevents corruption. Lost entry is recoverable.
**Considered and rejected:** Issue originally proposed step 5 as "run without config" — changed to fail-closed per user preference.
**Assumptions challenged:** "Should unregistered repos run with reduced checks?" — No, fail the gate.
**Risks accepted:** Concurrent register could lose an entry (re-register recovers).

## Design Approach

### Conceptual approach
New `config::registry` module handles all registry operations (CRUD, lookup, config home resolution). The existing `resolve_config_source()` function gains awareness of the project directory and calls into the registry module when `--config` and env var produce nothing.

Config home follows XDG conventions:
- `XDG_CONFIG_HOME/sloppy-joe/` if `XDG_CONFIG_HOME` is set
- macOS default: `~/Library/Application Support/sloppy-joe/`
- Linux default: `~/.config/sloppy-joe/`
- Legacy fallback: `~/.sloppy-joe/` if it exists and XDG location doesn't

### Codebase context
Config pipeline is clean: CLI → `resolve_config_source()` → `load_config_from_source()` → `scan_with_config()`. Registry slots in at the resolve step. `cache.rs` has atomic write patterns to reuse. `config.rs` is 775 lines — splitting into a module directory is justified.

### Thinking Record: Design Approach
**Decided:** New `config::registry` module, XDG config home with legacy fallback, convert `config.rs` to module directory.
**Reasoning:** Registry is a distinct concept from config loading — separate module keeps concerns clean. XDG follows platform conventions. Module split justified by file size + new concept.
**Alternatives rejected:** Inlining registry logic into `resolve_config_source()` (bloats existing function), hardcoded `~/.sloppy-joe/` only (doesn't follow platform conventions).
**Codebase findings:** Config pipeline is clean and modular. `cache.rs` atomic write patterns are reusable.
**Tradeoffs accepted:** Two possible config home locations (XDG + legacy) adds small complexity.

## Implementation Design

### Architecture

**Module structure:**
- `src/config/mod.rs` — existing `config.rs` content (moved), re-exports everything
- `src/config/registry.rs` — registry CRUD, lookup, config home resolution, git root detection

**Data model — `registry.json`:**
```json
{
  "/Users/brenn/dev/sloppy-joe": "/Users/brenn/.config/sloppy-joe/sloppy-joe/config.json",
  "/Users/brenn/dev/upfront": "/Users/brenn/.config/sloppy-joe/upfront/config.json"
}
```
`HashMap<String, String>` — canonicalized git root → canonicalized config path.

**New CLI commands:**
- `Register { dir: PathBuf, config: Option<String> }` — register a repo → config mapping
- `Unregister { dir: PathBuf }` — remove a repo from registry
- `List` — show all registered repos and their config paths

**Updated `Init`:**
- `--register` flag — creates config under config home AND registers cwd
- `--global` flag — creates `default/config.json` under config home

**Config resolution (updated `resolve_config_source`):**
1. `--config` flag
2. `SLOPPY_JOE_CONFIG` env var
3. `registry::lookup(git_root)` — find git root, match against registry
4. `{config_home}/default/config.json` if it exists
5. `None` (caller errors: "No config found")

**Key functions in `config::registry`:**
- `config_home() -> PathBuf` — XDG + legacy resolution
- `find_git_root(dir: &Path) -> Option<PathBuf>` — walk up looking for `.git`
- `load_registry() -> Result<HashMap<String, String>>` — read + parse registry.json
- `save_registry(entries: &HashMap<String, String>) -> Result<()>` — atomic write
- `register(repo_root: &Path, config_path: &Path) -> Result<()>` — add entry
- `unregister(repo_root: &Path) -> Result<()>` — remove entry
- `lookup(project_dir: &Path) -> Result<Option<String>>` — find git root, check registry, check default

### Structural issues flagged
None — config pipeline is clean, registry slots in naturally.

### Constraints and Non-negotiables
- Security: `ensure_config_outside_project` must apply to registry-resolved paths, not just `--config` paths
- Security: Registry-resolved config paths must be canonicalized before use
- Reliability: All failures are blocking errors (fail closed), never silent fallback

### Known AI Blind Spots
- [x] Edge cases: Path canonicalization — trailing slashes, symlinks, case sensitivity on macOS. Must canonicalize at both write and read time.
- [N/A] Concurrency: Atomic write sufficient for single-user tool.
- [x] Error handling: `ensure_config_outside_project` must fire on registry-resolved paths. Registry file missing vs corrupted are different errors with different messages.
- [x] Security: Registry is trusted input from a writable location. Acceptable because CI doesn't use it.
- [x] Non-prompted concerns: macOS `Library/Application Support` has a space — path handling in tests and display. `init --register` without `--register` must still just print to stdout (backward compat).
- [N/A] Hallucination risk: No external packages or APIs involved.

### Rollback Plan
- Trigger signal: `sloppy-joe check` fails in repos that worked before
- Method: Revert commit. Feature is additive — existing `--config` and env var paths are unchanged.
